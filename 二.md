我这次还没把 `rules` 目录里的每个 `.cypher` 文件逐条展开到文件级，但从你现在这条扫描链里，**rule 层该怎么改已经很清楚了**：增强阶段会先删掉旧的 `SRC_FLOW` 再重建一整层新的 `SRC_FLOW`，里面有 `assign_contains`、`receiver`、`interproc_param`、`compose2var`、`compose2call`、`interproc_call` 等高扇出边；规则执行器则对每个 Cypher 文件直接 `session.run(...)`，随后 `list(result)` 整批拉回结果；而聚合阶段判断 finding 时，本质上只看 `rule_rows > 0` 是否命中。换句话说，你现在最贵的并不是“漏洞是否存在”的判定，而是“为了导出很多条路径证据而做的全量枚举”。 ([GitHub](https://github.com/hjjing123/CodeScope/tree/main/backend))

这也解释了你列的三个难点为什么会同时出现。规则精度，主要取决于 source/sink/sanitizer 锚点定义得准不准；路径类规则的召回，主要取决于是否还能跨过程、跨间接传播链连通；性能爆炸，则主要来自把增强后的强边和弱边混在一条长路径里一起枚举。你自己的路径展示层其实已经证明“不需要在 rule 阶段把所有路径都吐出来”：`finding_path_service` 默认就用 `shortestPath(...)` 或按长度排序后 `LIMIT`，语义路径也限制在 `[:REF|ARG|PARAM_PASS|SRC_FLOW*..20]`，并且总条数被压到最多 20。 ([GitHub](https://github.com/hjjing123/CodeScope/blob/main/backend/app/services/finding_path_service.py))

所以，**不改图增强时，rule 层最稳的优化方向只有一个：把规则从“检测 + 全路径枚举”改成“三段式”**。

第一段是**锚点缩圈**。先用规则自己的语义把 sink/source 缩到很小的候选集，再做遍历。不要一上来就 `MATCH p=(src)-[:...*]->(sink)`。Neo4j 官方文档说明，最短路查询在 source/target 基数很低时会走更快的 shortest-path 规划；source-target 对很多时，会退化到成本更高的 `StatefulShortestPath(All)`。`CALL` 子查询也正适合把搜索限制在单个 sink 或单个 source 的小作用域里。 ([Graph Database & Analytics](https://neo4j.com/docs/cypher-manual/current/subqueries/call-subquery/))

第二段是**reachability 判断**，也就是只问“有没有路”，先别急着把整条 `p` 取出来。Neo4j 的 `EXISTS` 子查询就是干这个的：判断某个模式是否至少存在一次，而不是把所有路径结果都返回。对你这套系统，这一步最重要，因为 orchestrator 最终只需要知道这条 rule 有没有命中，根本不需要在这一阶段就拿到几十上百条等价路径。 ([Graph Database & Analytics](https://neo4j.com/docs/cypher-manual/current/subqueries/existential/))

第三段才是**代表路径回填**。也就是只对已经命中的 `(source, sink)` 候选对，补 1 条最短路径，或者很少量的代表路径。这一点跟你现有的 `finding_path_service` 是一致的：它本来就支持按 source/sink 的 file+line 或 node ref 去回查最短语义路径。也就是说，规则阶段完全可以只返回 `source_node_ref` / `sink_node_ref` 这类 canonical case，真正的易追踪路径延后到查询展示阶段再补。这样不会动图增强，也不会伤到命中精度；变化的只是“规则阶段一次性导出多少备选路径”。 ([GitHub](https://github.com/hjjing123/CodeScope/blob/main/backend/app/services/finding_path_service.py))

你现在的增强脚本里，`SRC_FLOW` 其实已经被分成了不同强度的语义摘要边。这一点特别关键。按“可达性放大风险”来看，我会把 `interproc_param`、`interproc_call`、`call_assign`、`ctor_assign`、`receiver` 看成偏强语义边；把 `assign_contains`、`temp_assign`、`compose2var`、`compose2call` 看成更容易放大搜索空间的边。这里不是说弱边不能用，而是**不要在全图搜索阶段和强边一起无差别地混跑**。 ([GitHub](https://github.com/hjjing123/CodeScope/raw/refs/heads/main/backend/app/services/scan_external/source_semantic_enhance.py))

最好的 rule 改法是做**双层边策略**。全局检测阶段，只允许强边参与 reachability；如果命中，就直接收敛到 source/sink pair。只有在强边阶段没命中、但这条规则又确实依赖间接传播链时，再对已经缩小过的 anchors 跑一轮弱边 fallback。这样做的意义是：全局搜索不再被 `assign_contains`/`compose2*` 这类边拖爆，但召回不会直接丢，因为弱边不是删掉，而是延后到更小的候选空间里用。支撑这个策略的前提，正是你增强图已经把 `interproc_*` 和 `compose2*` 这类摘要边显式物化出来了。 ([GitHub](https://github.com/hjjing123/CodeScope/raw/refs/heads/main/backend/app/services/scan_external/source_semantic_enhance.py))

还有一个很容易被忽视的点：**既然增强层已经给你做了摘要边，规则就不要再同时走“摘要边 + 底层长链分解”两套表示。** 例如跨过程传播，如果规则已经可以借 `interproc_param` / `interproc_call` 连过去，就不要又在同一条查询里重新走一大段 `[:REF|ARG|PARAM_PASS|SRC_FLOW*..N]` 去把这段过程间传播再展开一遍；两种表示叠加，会制造大量等价路径。两跳组合也是一样，`compose2var` / `compose2call` 已经是压缩后的两跳摘要，如果规则还把底层两跳和摘要边一起混用，路径数会成倍放大。这个优化不改图，只改 rule 的“走哪种边表示”。 ([GitHub](https://github.com/hjjing123/CodeScope/raw/refs/heads/main/backend/app/services/scan_external/source_semantic_enhance.py))

你还可以顺手做一个**几乎不改规则语义、但对性能很值的补丁**：把 `SRC_FLOW.kind` 的索引补回来。增强脚本当前可见语句里明确有 `DROP INDEX srcflow_kind IF EXISTS`，但在这份可见代码里我没看到对应的重建；而 Neo4j 是支持关系属性索引的。如果你的规则准备开始显式按 `sf.kind` 分层筛边，这个索引非常值得保留。这个不属于改图增强语义，只是运行时 schema 优化。 ([GitHub](https://github.com/hjjing123/CodeScope/raw/refs/heads/main/backend/app/services/scan_external/source_semantic_enhance.py))

你可以直接把路径类 rule 按下面这个骨架改：

```cypher
// A. 先缩 sink anchors
MATCH (sink:Call)
WHERE ...  // 这条规则自己的危险 sink 条件

CALL {
  WITH sink
  // B. 再缩 source anchors
  MATCH (src:Var)
  WHERE ...  // 这条规则自己的 source / sanitizer / scope 条件

  // C. 这里只做 existence / reachability
  WITH src, sink
  WHERE EXISTS {
    MATCH (src)-[:REF|ARG|PARAM_PASS|SRC_FLOW*..N]->(sink)
    WHERE ... // 阻断条件、版本条件、局部范围条件
  }

  RETURN DISTINCT src.id AS src_id
  LIMIT 1
}

WITH DISTINCT src_id, sink.id AS sink_id

// D. 只对命中的 pair 回填一条代表路径
MATCH (src {id: src_id}), (sink {id: sink_id})
MATCH p = shortestPath((src)-[:REF|ARG|PARAM_PASS|SRC_FLOW*..N]->(sink))
RETURN src_id, sink_id, p
```

这个骨架不要求你放弃增强图，只是把增强图的使用顺序改了：先用它判断能不能到，再用它补代表路径，而不是一开始就在增强图上把所有可能路径都展开。它跟你现有的聚合逻辑和路径展示逻辑是兼容的。 ([GitHub](https://github.com/hjjing123/CodeScope/blob/main/backend/app/services/scan_external/orchestrator.py))

如果按“投入最小、收益最大”排优先级，我建议你先做这四件事。先把所有路径类 rule 的返回粒度改成 `DISTINCT source/sink pair`，不要按 path 粒度返回。再把跨过程规则改成优先走 `interproc_param` / `interproc_call` 这类摘要边。然后把弱边 `assign_contains` / `compose2*` 改成 fallback，而不是默认主路径。最后，把 `SRC_FLOW.kind` 的关系索引补回去。这样基本不碰图增强本身，但会明显降低增强图对规则阶段的放大效应。 ([GitHub](https://github.com/hjjing123/CodeScope/raw/refs/heads/main/backend/app/services/scan_external/source_semantic_enhance.py))
