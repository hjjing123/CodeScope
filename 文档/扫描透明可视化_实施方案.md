# 扫描透明可视化_实施方案

- 文档版本：v0.1.1
- 状态：Draft
- 最后更新：2026-03-08

## 1. 背景

当前平台已具备扫描任务、阶段日志、external 四段扫描编排等基础能力，但对用户而言，扫描过程仍偏“黑盒”：

- 用户能看到任务状态，但不容易知道当前卡在哪个扫描子流程。
- 用户能查询日志尾部，但缺少一个持续滚动、接近终端体验的实时输出页面。
- 扫描任务进入 `RUNNING` 后，前端侧缺少统一的“流程进度 + 实时输出”联动展示。

本方案聚焦“扫描透明可视化”，先解决过程可见、进度可感知、输出可追踪的问题，不扩大到结果研判和证据链详情。

## 2. 我的需求

用户明确提出以下需求：

1. 在前端有明确的流程进度划分。
2. 创建扫描任务后，可以点击任务进入一个类似终端的页面，看到实时的扫描输出过程。

将其转成产品目标后，等价为：

- 用户在任务列表和任务详情中，能明确知道任务当前位于哪一个扫描步骤。
- 用户进入任务详情页后，能持续看到扫描过程中的新增输出，而不是只在任务结束后回看日志。
- 用户看到的进度与日志内容要互相对应，避免“进度显示在 A，日志其实跑到 B”。

## 3. 目标与边界

### 3.1 目标

- 将当前 external 扫描链路拆解为前端可直接展示的步骤流。
- 提供任务详情页的“终端式实时日志面板”。
- 让前端无需解析自由文本日志，也能渲染扫描步骤状态。
- 保留现有日志查询与下载能力，避免推翻已有实现。

### 3.2 边界

- 本阶段只覆盖扫描任务，不扩展到导入任务、报告任务、AI 任务。
- 本阶段优先支持当前 external 扫描流程：`joern -> neo4j_import -> post_labels -> rules`。
- 本阶段不做复杂图形化拓扑展示，不展示 Neo4j 内部图结构预览。
- 本阶段不要求 WebSocket 为唯一方案，允许先用 SSE 落地实时流。

## 4. 现状审查

### 4.1 当前已有能力

- 扫描任务具备主阶段：`PREPARE / ANALYZE / QUERY / AGGREGATE / CLEANUP`。
- external 扫描具备子阶段：`joern / neo4j_import / post_labels / rules`。
- 任务日志已支持查询与下载。
- 任务执行结果中已存在 `external_stages` 结构，可作为可视化基础。

### 4.2 当前不足

- 主阶段粒度过粗，前端无法直接展示 external 四段细分流程。
- 任务详情日志更偏“查询 tail”，不是实时推送。
- 进度信息主要散落在日志文本和 `result_summary` 中，缺少运行中可稳定查询的结构化步骤状态。
- 若任务尚未结束，前端难以获得完整的“当前步骤、已完成步骤、失败步骤、耗时”信息。

## 5. 计划

分三期推进，先打通最小闭环，再增强体验。

### 5.1 第一期：结构化步骤可视化

- 为扫描任务补充运行中步骤状态输出。
- 前端任务详情页展示固定流程步骤条。
- 任务列表支持展示“当前步骤名称”。

### 5.2 第二期：终端式实时日志

- 提供扫描日志实时流接口。
- 前端任务详情页新增终端面板，实时滚动显示日志。
- 支持按阶段过滤日志输出。

### 5.3 第三期：体验增强

- 支持自动滚动开关、日志关键字高亮、错误快速定位。
- 支持显示每个步骤耗时、开始时间、结束时间。
- 支持失败时自动定位到最后错误片段。

## 6. 方案

### 6.1 流程进度方案

前端统一展示以下扫描步骤：

1. 准备源码
2. Joern 解析生成 CPG
3. Joern 导出图 CSV
4. 导入 Neo4j
5. 图增强 post_labels
6. 规则扫描
7. 结果聚合
8. 清理归档

说明：

- 第 1、7、8 步对应主阶段。
- 第 2 到第 6 步对应 external 子阶段。
- 前端不直接依赖日志文案推断步骤，而是使用后端返回的结构化状态。

每个步骤统一状态：

- `pending`：未开始
- `running`：执行中
- `succeeded`：已完成
- `failed`：失败
- `skipped`：跳过

每个步骤最少返回字段：

- `key`
- `name`
- `order`
- `status`
- `started_at`
- `finished_at`
- `duration_ms`
- `message`

建议新增任务详情结构字段：

```json
{
  "progress": {
    "current_step": "neo4j_import",
    "completed_steps": 3,
    "total_steps": 8,
    "percent": 37
  },
  "steps": [
    {
      "key": "prepare",
      "name": "准备源码",
      "order": 1,
      "status": "succeeded",
      "started_at": "2026-03-08T10:00:00Z",
      "finished_at": "2026-03-08T10:00:02Z",
      "duration_ms": 2000,
      "message": "源码快照已就绪"
    }
  ]
}
```

落地建议：

- 后端在任务运行中持续维护步骤状态，而不是只在结束后汇总到 `result_summary`。
- `jobs` 查询接口返回 `progress` 和 `steps` 字段。
- `result_summary.external_stages` 继续保留，作为归档摘要；`steps` 作为运行态展示模型。

### 6.2 后端状态建模方案

建议在当前实现上增加一个“运行态步骤快照”存储层，优先选择以下两种方式之一：

- 方案 A：写入 `jobs.result_summary_runtime` 或 `jobs.runtime_meta` 之类的 JSON 字段。
- 方案 B：新增 `job_steps` 表，按任务维度记录每一步的状态与耗时。

推荐优先采用方案 B，原因：

- 结构清晰，便于前端直接消费。
- 支持运行中频繁更新，不会不断覆盖整块 JSON。
- 后续可扩展到导入任务、报告任务、AI 任务。

`job_steps` 建议字段：

- `id`
- `job_id`
- `step_key`
- `step_name`
- `step_order`
- `status`
- `started_at`
- `finished_at`
- `duration_ms`
- `message`
- `error_code`
- `error_message`
- `created_at`
- `updated_at`

状态更新时机建议：

- 进入某一步时：写 `running`
- 成功结束时：更新为 `succeeded`
- 失败时：更新为 `failed`
- 被配置跳过时：更新为 `skipped`

### 6.3 任务详情终端页方案

新增一个扫描任务详情页区域：`实时输出`。

页面形态：

- 深色背景，等宽字体。
- 按时间顺序滚动追加日志。
- 默认自动滚动到底部。
- 显示时间戳、阶段、日志正文。
- 支持阶段过滤：`ALL / PREPARE / ANALYZE / QUERY / AGGREGATE / CLEANUP`。

推荐交互：

- 任务运行中时，右上角显示 `LIVE`。
- 用户可关闭自动滚动后手动查看历史。
- 任务失败时自动高亮最后一个错误片段。
- 任务成功后保留完整日志回放能力。

示例展示形态：

```text
[10:00:02] [PREPARE] 源码快照已挂载
[10:00:03] [ANALYZE] [joern] 执行 parse 命令...
[10:00:21] [ANALYZE] [joern] 导出完成: nodes=12 relationships=18
[10:00:22] [ANALYZE] [neo4j_import] 执行导入命令...
[10:00:39] [QUERY] [post_labels] 执行完成: statements=8 total_rows=134
[10:00:42] [QUERY] [rules] 执行完成: total_rules=12 hit_rules=2
```

### 6.4 实时日志传输方案

建议优先采用 SSE，暂不强制 WebSocket。

原因：

- 该场景主要是服务端单向推送日志，SSE 更轻。
- 接入成本低，前后端实现复杂度小。
- 对“终端式滚动输出”已经足够。

建议新增接口：

- `GET /api/v1/jobs/{job_id}/logs/stream`

SSE 事件结构建议：

```json
{
  "job_id": "xxx",
  "stage": "ANALYZE",
  "step_key": "joern_export",
  "timestamp": "2026-03-08T10:00:21Z",
  "line": "[joern] 导出完成: nodes=12 relationships=18",
  "seq": 128
}
```

后端行为建议：

- 任务运行中：日志写文件的同时推送 SSE。
- 前端断线重连：通过 `Last-Event-ID` 或 `seq` 补拉丢失日志。
- 任务结束后：SSE 自动推送 `done` 事件并关闭。

降级策略：

- 若 SSE 不可用，前端退化为轮询 `GET /api/v1/jobs/{job_id}/logs?after_seq=...`。

### 6.5 与当前扫描实现的对接点

需要改造的关键点：

- `scan_service`
  - 负责初始化固定步骤列表。
  - 在主阶段切换时同步更新步骤状态。
- `scan_external.orchestrator`
  - 在 `joern / neo4j_import / post_labels / rules` 开始和结束时更新对应子步骤。
  - 将失败信息写入步骤状态，而不只是抛异常。
- 任务日志写入组件
  - 在现有写日志基础上增加“实时事件发布”。
- 前端任务详情页
  - 新增步骤条组件。
  - 新增终端日志组件。

### 6.6 前端展示方案

任务详情页建议拆为上下两块：

- 上半部分：扫描进度
  - 步骤条
  - 当前步骤文案
  - 累计耗时
  - 失败时展示失败步骤和错误摘要
- 下半部分：实时输出
  - 终端面板
  - 阶段过滤
  - 自动滚动开关
  - 清空本地视图按钮（不影响服务端日志）

任务列表页建议补充：

- 当前步骤标签，如“导入 Neo4j 中”
- 任务运行中时的动态状态点

## 8. 按扫描流程逐段优化方案

本节按实际扫描流程逐段讨论优化方向。原则是每一段同时关注三件事：

- 执行稳定性
- 性能与复用
- 过程透明可视化

### 8.1 第 1 段：源码准备与进入扫描

目标：让用户在点击“开始扫描”后，立刻感知任务已启动，并确保本次扫描拥有独立、可追踪的执行上下文。

#### 当前问题

- 用户创建任务后，前端只能看到任务进入 `RUNNING`，但不知道系统具体正在做什么。
- 任务前几秒主要是平台内部动作，若页面没有实时系统日志，用户容易误以为任务卡住。
- 当前 external 工作区按 `project_id/version_id` 组织，重试任务或并发任务存在相互覆盖风险。

#### 优化点

- 工作区按 `job_id` 隔离。
  - 每个扫描任务创建独立工作目录。
  - 避免同一版本的重试、并发、补扫任务共用 `code.bin`、`import_csv` 等中间产物。
- 将准备阶段拆成结构化子动作。
  - 建议拆分为：
    - `prepare.validate_input`
    - `prepare.create_workspace`
    - `prepare.load_snapshot`
    - `prepare.check_runtime`
- 任务创建后立即写入第一批系统事件日志。
  - 即使外部命令尚未启动，终端页也应立即有输出。
- 前端流程条将第一步固定展示为 `准备源码`。
  - 步骤高亮不再只是 `RUNNING`，而是明确显示当前处于准备阶段。

#### 可视化方案

- 前端步骤条：高亮 `准备源码`。
- 任务详情页终端面板立即输出：

```text
[10:00:00] [SYSTEM] 扫描任务已创建 job_id=...
[10:00:00] [PREPARE] 开始校验项目与版本
[10:00:01] [PREPARE] 创建任务工作区成功
[10:00:01] [PREPARE] 源码快照已加载
[10:00:02] [PREPARE] 扫描配置校验完成
```

- 前端当前步骤文案可显示：`正在准备源码`。

#### 实施建议

- 后端在任务启动时初始化固定步骤列表。
- 在 `prepare` 阶段内记录结构化步骤状态与系统日志。
- 调整工作区路径策略，将 job 维度纳入路径。

#### 本段验收

- 创建扫描任务后，详情页在 1 秒内出现首批系统输出。
- 同一 `version_id` 下连续创建两个扫描任务时，工作区互不覆盖。
- 用户能够明确看到当前处于 `准备源码`，而不是仅看到笼统的 `RUNNING`。

## 9. 风险与注意事项

- 日志量过大时，实时推送和前端渲染可能造成卡顿，需要做窗口化或最大缓存行数限制。
- 日志中不得输出密码、令牌、连接串凭据等敏感信息，继续沿用脱敏约束。
- 步骤状态必须以后端结构化状态为准，不能由前端根据日志文本猜测。
- 若后端先写日志再更新步骤状态，前端可能出现短暂不同步，需要定义更新顺序。

## 10. 验收

### 9.1 功能验收

- 创建扫描任务后，任务详情页能看到固定的扫描步骤列表。
- 任务运行时，当前步骤会自动高亮，并随执行推进。
- 点击任务后可进入终端式页面，实时看到新增日志输出。
- 任务失败时，页面能显示失败步骤和最后错误片段。
- 任务完成后，用户仍可回看完整日志。

### 9.2 接口验收

- `GET /api/v1/jobs/{job_id}` 返回结构化步骤信息。
- `GET /api/v1/jobs/{job_id}/logs/stream` 可持续推送新增日志。
- SSE 中断后可恢复，不会产生明显日志丢失。

### 9.3 体验验收

- 用户无需刷新页面即可感知扫描推进。
- 用户能明确区分“卡在 Joern”“卡在 Neo4j 导入”“卡在规则扫描”。
- 实时日志展示延迟控制在可接受范围内，目标为 1 秒内可见。

## 11. 里程碑建议

- M1：结构化步骤状态打通，前端可显示流程进度。
- M2：SSE 实时日志打通，前端可显示终端式实时输出。
- M3：失败定位、自动滚动、断线重连、性能优化完善。
