有，而且按你现在这套架构，**不改 rule 文件**也能拿到一波很实在的性能收益。关键点不是再去抠 Cypher，而是把控制点往 **执行器、Neo4j 运行时、结果处理层** 挪。因为你现在的代码里，`source_semantic_enhance.py` 会重建一整层 `SRC_FLOW`，`neo4j_runner.py` 又是 `session.run(statement)` 后直接 `list(result)` 全量拉回；同时 orchestrator 生成 finding 时，本质上只看 `rows > 0` 是否命中，路径展示则已经有单独、带上界的 `finding_path_service`。这说明你有很大空间在 **不动 rule 文本** 的前提下，把“全量枚举”改成“受控执行 + 延迟取证”。 ([GitHub](https://github.com/hjjing123/CodeScope/raw/refs/heads/main/backend/app/services/scan_external/source_semantic_enhance.py))

我按“收益/改动比”给你排一下。

### 1. 先改 `neo4j_runner.py`：去掉 `list(result)`，改成流式/分块处理

这是最该先做的。你现在在 `execute_cypher_file_stream()` 里先 `session.run(statement)`，然后 `raw_records = list(result)`，接着再 `postprocess_result_records(...)`。Neo4j Python Driver 文档明确说，`Result` 会在客户端维护缓冲区，未消费的数据会被缓冲；结果对象会一直绑定连接，直到被完全缓冲或消费。也就是说，你现在的写法把数据库端的大结果集和 Python 端的大对象堆在了一起。改成“边拉边处理”或“每 100/200 条做一次 postprocess”的分块模式，不改 rule 语义，却能明显降低内存峰值和长尾抖动。`fetch_size` 默认是 1000；只有在你不再 `list(result)` 之后，调小到 100~500 才真正有意义。 ([GitHub](https://github.com/hjjing123/CodeScope/raw/refs/heads/main/backend/app/services/scan_external/neo4j_runner.py))

你可以直接把 runner 改成这个形态：

```python
from neo4j import Query

with driver.session(database=database, fetch_size=200) as session:
    for idx, statement in enumerate(statements, 1):
        q = Query(
            statement,
            metadata={"stage": "rules", "rule_file": cypher_file.name, "stmt": idx},
            timeout=per_stmt_timeout_s,
        )
        result = session.run(q)
        keys = list(result.keys())
        if not keys:
            result.consume()
            row_counts.append(0)
            continue

        buf = []
        row_count = 0
        for raw in result:
            buf.append(raw)
            if len(buf) >= 200:
                records, _ = postprocess_result_records(buf, keys, path_post_cfg)
                for record in records:
                    row_count += 1
                    if on_record is not None:
                        on_record(_serialize_record(record))
                buf.clear()

        if buf:
            records, _ = postprocess_result_records(buf, keys, path_post_cfg)
            for record in records:
                row_count += 1
                if on_record is not None:
                    on_record(_serialize_record(record))

        result.consume()
        row_counts.append(row_count)
```

如果 `postprocess_result_records()` 必须看整批数据，就把它改成“分块安全”；这是执行器改动，不是 rule 改动。

### 2. 给每条 rule 加数据库级 timeout，而不是只靠 stage timeout

你当前 driver 只设置了 `connection_timeout=5`。Neo4j 官方文档写得很明确：`connection_timeout` 只是 TCP 建连超时，不是查询执行超时；真正控制查询/事务执行时间的是 `Query(..., timeout=...)` / transaction timeout，以及数据库端的 `db.transaction.timeout`。你 README 里虽然已经暴露了 `CODESCOPE_SCAN_EXTERNAL_STAGE_RULES_TIMEOUT_SECONDS`，但源码里的 rule runner 目前没有把这个预算传给 Neo4j 本身。结果就是：应用层知道“这阶段超时了”，数据库层却可能还在硬跑那条爆炸遍历。 ([GitHub](https://github.com/hjjing123/CodeScope/raw/refs/heads/main/backend/app/services/scan_external/neo4j_runner.py))

这一步不改变 rule 精度本身；它只是把“拖死整个规则阶段”的异常情况，变成“单条 rule 受控失败”。为了尽量不影响召回，你可以先用保守预算，比如：

- 开发机先给单条 rule 15~30 秒；
- 再把它和现有的 `CODESCOPE_SCAN_EXTERNAL_STAGE_RULES_TIMEOUT_SECONDS` 联动；
- 只对超长尾 rule 生效，正常规则不受影响。

同时把 `metadata={"stage":"rules","rule_file":...}` 带进去。Neo4j driver 支持把 metadata 挂到 transaction 上，能出现在 `SHOW TRANSACTIONS` 和 `query.log` 里，后面定位慢规则会轻松很多。 ([Graph Database & Analytics](https://neo4j.com/docs/api/python-driver/current/api.html))

### 3. 不改 rule 文件，也可以做“命中优先、证据延后”

这招很适合你现在的代码结构。因为 orchestrator 里，rule 最终是否变成 finding，当前判断条件就是 `rows > 0`；而路径展示这件事，你已经放在 `finding_path_service.py` 里做了，且这层查询本身就是受控的：`shortestPath`、`*..20`、`LIMIT`。这意味着 rule 阶段其实没必要把大量路径证据一次性全吐出来。 ([GitHub](https://github.com/hjjing123/CodeScope/blob/main/backend/app/services/scan_external/orchestrator.py))

这里有两个不改 rule 文件的落地方式：

第一种更稳：**rule 阶段照常执行，但只持久化最小命中信息**，例如 `rule_key`、首尾节点 ref、文件行号；完整路径留到用户点开 finding 时，再让 `finding_path_service` 反查。
第二种更激进：**runner 做双阶段执行**。第一遍对 read-only rule 做 detect-only 包装，只判断有没有命中；第二遍只对 hit 的 rule 取证。这不改 rule 文件，但会改 runner 的执行策略，兼容性要先回归一下。

第一种我更推荐，因为它不会碰查询语义，只是减小 Python/MinIO/网络传输的压力。

### 4. 结果序列化瘦身：别在 rule 阶段无脑把整条 path 全序列化

`neo4j_runner.py` 现在的 `_serialize_graph_value()` 会把 path 里的全部 nodes / edges 都摊平成 JSON。对路径型结果，这个成本很高，尤其当你的规则已经建立在增强图上、path 边数又多的时候。可你后面本来就有 `finding_path_service` 去回查语义最短路，所以 rule 阶段其实不一定要把整条路径全部存下来。 ([GitHub](https://github.com/hjjing123/CodeScope/raw/refs/heads/main/backend/app/services/scan_external/neo4j_runner.py))

这个优化对**数据库遍历时间**帮助有限，但对**总耗时、内存、对象存储写入、前后端传输**很有用。你可以把 rule 阶段的 path 存储降到：

- path 起点 node_ref
- path 终点 node_ref
- path 长度
- 少量 edge type 摘要

真正需要可视化时，再从增强图里回查一条代表路径。

### 5. Neo4j 运行时单独调：给 rules 阶段更硬的边界

你的 README 已经有 task-scoped runtime、`docker_ephemeral`、`CODESCOPE_SCAN_EXTERNAL_RUNTIME_MAX_SLOTS`、rules stage timeout 这些配置；这说明你本来就支持把扫描跑在独立 Neo4j runtime 上。既然如此，最值钱的做法就是把 rules 阶段当成**内存型、路径型工作负载**去配，而不是沿用默认值。Neo4j 官方配置里，`db.transaction.timeout` 可以限制事务最长执行时间，`dbms.memory.transaction.total.max` 可以限制所有运行中事务的总内存，`server.memory.heap.*` 和 `server.memory.pagecache.size` 则控制堆和页缓存。 ([GitHub](https://raw.githubusercontent.com/hjjing123/CodeScope/main/backend/README.md))

对你这类 workload，我建议：

- **rules 阶段并发降到 1**，至少先把 `CODESCOPE_SCAN_EXTERNAL_RUNTIME_MAX_SLOTS` 调低，避免多条路径遍历同时吃事务内存；
- 给 scan 专用 Neo4j 容器单独配堆和 page cache；
- 打开有限的 `db.transaction.timeout` 和 `dbms.memory.transaction.total.max`，把“爆炸”变成“熔断”。

这不会改变 rule 精度；它改变的是系统在坏情况出现时的退化方式。代价是并发少了，但通常总吞吐会更稳定。

### 6. 只补 schema，不改增强语义：把 `SRC_FLOW.kind` 索引补回来

`source_semantic_enhance.py` 一开始会 `DROP INDEX srcflow_kind IF EXISTS`，然后删掉全部 `SRC_FLOW` 再重建；但当前可见代码里没有把这个索引重新建回来。更关键的是，增强脚本自己最后一条 `interproc_call` 就已经写了 `sf.kind = 'interproc_param'` 这样的过滤条件。Neo4j 官方文档支持对 relationship property 建 range index，而且规划器会自动选择可用索引。 ([GitHub](https://github.com/hjjing123/CodeScope/raw/refs/heads/main/backend/app/services/scan_external/source_semantic_enhance.py))

所以这一步其实属于**非语义性补丁**：

```cypher
CREATE INDEX srcflow_kind IF NOT EXISTS
FOR ()-[r:SRC_FLOW]-() ON (r.kind)
```

如果你坚持完全不碰增强逻辑，那也至少在 post-labels / rules 之前补一段 schema 初始化，把这个索引建回来。它不会改变图，不会改规则，只是给后面依赖 `sf.kind` 的过滤留优化空间。
我这里要实话实说：**只有当规则或后续查询真的用到 `sf.kind` 过滤时，这个索引才有明显收益**。但你的增强脚本自己已经有这类过滤了，所以它不是空枪。 ([GitHub](https://github.com/hjjing123/CodeScope/raw/refs/heads/main/backend/app/services/scan_external/source_semantic_enhance.py))

### 7. 先把慢点看清楚：给 rule runner 加 statement 级耗时日志

你现在的 `source_semantic_enhance.py` 已经会记录每条增强语句的 `duration_ms`，但 `neo4j_runner.py` 还没有对 rule statement 做同等级别的耗时和 counters 记录。Neo4j 的 query log 也支持按阈值记录慢查询；如果是 Enterprise，还能记 transaction/query 级日志，metadata 也会跟着进日志。 ([GitHub](https://github.com/hjjing123/CodeScope/raw/refs/heads/main/backend/app/services/scan_external/source_semantic_enhance.py))

这一步本身不提速，但它能告诉你：

- 到底是数据库遍历慢，还是 Python 序列化慢；
- 是哪几条 rule 最值得进入“detect-only / 延迟取证”名单；
- 你的 timeout 该设在 10 秒、30 秒还是 60 秒。

## 我最推荐的最小落地组合

如果你现在只想做**最少改动**，我建议先做这四件事：

1. `neo4j_runner.py` 去掉 `list(result)`，改成分块流式处理。 ([GitHub](https://github.com/hjjing123/CodeScope/raw/refs/heads/main/backend/app/services/scan_external/neo4j_runner.py))
2. 用 `Query(timeout=..., metadata=...)` 或 transaction timeout，把单条 rule 的运行时间交给 Neo4j 控制。 ([Graph Database & Analytics](https://neo4j.com/docs/api/python-driver/current/api.html))
3. rules 阶段并发先降到 1，并给 scan 专用 Neo4j 容器配事务内存上限。 ([GitHub](https://raw.githubusercontent.com/hjjing123/CodeScope/main/backend/README.md))
4. rule 阶段不再持久化完整 path，只存最小命中信息，路径展示继续走你现有 `finding_path_service`。 ([GitHub](https://github.com/hjjing123/CodeScope/blob/main/backend/app/services/scan_external/orchestrator.py))

这四步里，前两步最直接，后三步最稳。
